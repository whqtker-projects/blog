## IoC란

IoC는 Inversion of Control의 약자입니다. 보통 **제어의 역전**이라고 부릅니다.

일반적으로 객체지향 코드를 작성할 때는 필요한 객체를 직접 만들고, 필요한 곳에 직접 연결합니다. 예를 들어 `OrderService`가 `OrderRepository`를 사용해야 하면 `OrderService` 내부에서 `OrderRepository` 구현 객체를 직접 생성합니다. 이런 구조에서는 객체의 생성, 연결, 사용 과정을 애플리케이션 코드가 직접 관리합니다.

스프링은 **객체를 생성하고 관리하고 연결하는 일을 개발자가 아닌 스프링 컨테이너가 담당**합니다. 개발자는 객체를 어떻게 조립할지보다, 어떤 역할의 객체가 필요한지와 비즈니스 로직 자체에 더 집중할 수 있습니다.

## DI란

DI는 Dependency Injection의 약자입니다. 보통 **의존성 주입**이라고 부릅니다.

여기서 의존성이란 **어떤 객체가 자신의 로직을 수행하기 위해 다른 객체를 필요로 하는 관계**를 말합니다. 예를 들어 `OrderService`가 주문 정보를 저장하기 위해 `OrderRepository`를 사용한다면, `OrderService`는 `OrderRepository`에 의존한다고 할 수 있습니다.

서비스 클래스 안에서 저장소 구현체를 직접 생성하면 서비스는 특정 구현 클래스에 강하게 의존하게 됩니다. 즉, **의존성을 객체 내부에서 설정하면 특정 클래스에 강하게 연결**됩니다. 레포지토리 클래스를 바꾸기 어려워지고, 테스트할 때 mock 객체를 넣기도 번거로워집니다.

DI는 이런 문제를 줄이기 위해 **객체가 필요한 의존성을 스스로 만들지 않고, 외부에서 주입받아 사용**하도록 하는 방식입니다. 스프링에서는 이 주입 과정을 스프링 컨테이너가 처리합니다.

## 스프링 컨테이너의 빈 생성과 주입 과정  

1. POJO 형태로 클래스를 작성합니다.
2. 어떤 클래스를 빈으로 등록할지, 빈 사이의 관계를 어떻게 구성할지에 대한 정보를 스프링에 전달합니다.
3. 스프링 컨테이너는 이 정보를 바탕으로 **런타임에 빈을 생성**하고, 필요한 의존관계를 주입합니다.
4. 생성된 빈은 컨테이너가 관리하며, 애플리케이션은 필요할 때 이를 조회하여 사용합니다.

## DI 방식

스프링에서 의존성을 주입하는 방식은 크게 생성자 주입, 세터 주입, 필드 주입으로 나눌 수 있습니다. 세 방식 모두 외부에서 의존성을 주입한다는 점은 같지만, 의존성을 주입하는 시점과 방식에는 차이가 있습니다.

### 설정 방식에 따른 DI

명시적 DI는 개발자가 설정 코드를 직접 작성하는 방식입니다. 대표적으로 **`@Configuration` 클래스와 `@Bean` 메서드**를 사용합니다. 어떤 객체를 빈으로 등록할지, 어떤 빈을 어떤 빈에 주입할지를 설정 코드를 통해 알 수 있습니다. 또한 외부 라이브러리 객체처럼 소스 코드를 수정할 수 없는 클래스도 이러한 방식으로 빈 등록이 가능합니다. 단, 설정 코드가 별도로 필요하고, 빈 수가 많아지면 관리해야 할 코드도 늘어납니다.

묵시적 DI는 스프링이 빈 등록과 의존관계 연결을 자동으로 처리하는 방식입니다. 대표적으로 컴포넌트 스캔과 자동 주입을 사용합니다. 클래스에 **`@Component`, `@Service`, `@Repository`, `@Controller`** 같은 어노테이션을 붙이면 스프링이 이를 찾아 빈으로 등록합니다. 필요한 의존성도 주입 지점을 기준으로 자동으로 연결합니다. 설정 코드를 많이 작성하지 않아도 되지만, 빈이 어디에서 등록되었는지, 전체 의존관계가 어떻게 이어지는지는 명시적 DI보다 한눈에 파악하기 어렵습니다.

일반적으로 애플리케이션 내부에서 직접 작성한 서비스, 레포지토리, 컨트롤러는 묵시적 DI를 사용하는 경우가 많고, 외부 라이브러리 객체를 빈으로 등록하거나 객체 생성 과정을 코드로 드러내야 하는 경우에는 명시적 DI를 사용합니다.

### 생성자 주입

생성자 주입은 객체를 만들 때 생성자를 통해 의존성을 주입받는 방식입니다.

**객체를 생성하는 순간 필요한 의존성이 함께 전달**되기 때문에, 어떤 의존성이 반드시 필요한지 코드를 통해 바로 알 수 있습니다. 또한 필드를 `final`로 둘 수 있어, 객체가 만들어진 뒤 의존성이 바뀌지 않도록 만들 수 있습니다.

스프링에서는 생성자가 하나뿐이면 `@Autowired`를 생략해도 자동으로 주입합니다.

```java
@Component
public class OrderService {

    private final OrderRepository orderRepository;

    public OrderService(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }
}
```

### 세터 주입

세터 주입은 setter 메서드를 통해 의존성을 주입하는 방식입니다.

이 방식은 객체를 생성한 뒤 필요한 의존성을 주입하는 방식입니다. 따라서 객체가 만들어질 때 반드시 필요하지 않은 의존성이나, 선택적으로 주입해도 되는 경우에 사용할 수 있습니다.

다만 객체가 생성되는 시점에 필요한 의존성이 모두 주입된다고 보장하기는 어렵습니다. 세터 호출이 누락되면 의존성이 주입되지 않은 상태로 객체가 생성될 수 있고, 이러한 상태에서 메서드가 실행되면 오류가 발생할 수 있습니다.

```java
@Component
public class OrderService {

    private OrderRepository orderRepository;

    @Autowired
    public void setOrderRepository(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }
}
```

### 필드 주입

필드 주입은 필드에 직접 의존성을 주입하는 방식입니다.

실무에서는 보통 권장하지 않는 방식입니다. 생성자만 보고서는 어떤 의존성이 필요한지 파악하기 어렵고, 스프링 컨테이너 없이 객체를 직접 생성해 테스트하기도 불편합니다. 또한 `final` 필드를 사용할 수 없어, 객체 생성 이후에도 의존성이 바뀔 수 있는 구조가 됩니다.

```java
@Component
public class OrderService {

    @Autowired
    private OrderRepository orderRepository;
}
```

## 생성자 주입을 권장하는 이유

### 필수 의존성이 명확하게 드러난다

생성자 파라미터를 보면 해당 객체가 어떤 의존성을 가지는지 바로 알 수 있습니다.

### 불완전한 객체가 만들어질 가능성을 줄일 수 있다

생성자 주입은 필요한 의존성이 없으면 객체를 생성할 수 없습니다. 반면 세터 주입은 의존성이 주입되지 않은 상태에서도 객체가 먼저 생성될 수 있습니다.

### `final`을 사용할 수 있다

객체 생성 이후 의존성이 바뀌지 않는 구조는 객체 상태를 예측하기 쉽고, 코드를 더 안정적으로 유지할 수 있습니다.

### 테스트하기 쉽다

스프링 컨테이너를 띄우지 않아도 테스트 코드에서 객체를 직접 생성하고, 원하는 구현체를 넣어 검증할 수 있습니다.

### 순환 참조 같은 문제를 빨리 찾을 수 있다

서로가 서로를 의존하는 구조가 있으면 스프링 컨테이너가 빈을 생성하는 단계에서 순환 참조 문제를 발견할 수 있습니다. 다른 방식은 빈 생성 후 주입 과정에서 발견하게 됩니다.
