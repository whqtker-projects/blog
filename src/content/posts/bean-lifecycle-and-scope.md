## 빈 생명주기란

스프링 빈은 객체를 생성한 뒤 바로 사용되는 것이 아니라, 필요한 준비를 마친 뒤 사용되다가 종료 시점에 정리됩니다. 이 흐름을 빈 생명주기라고 합니다.

스프링 컨테이너는 **빈을 생성하고 의존관계를 주입한 뒤, 초기화 작업을 수행하며, 컨테이너가 종료될 때 소멸 작업**까지 처리합니다.

외부 API 클라이언트, 메시지 큐 연결 객체, 데이터베이스 관련 객체처럼 생성 직후에 추가 설정이 필요한 경우가 있습니다. 또한 애플리케이션이 종료될 때 연결을 닫거나 자원을 반납해야 하는 경우도 있습니다. 이런 작업은 보통 초기화 단계와 소멸 단계에서 처리합니다.

![[Pasted image 20260513162746.png]]

빈 생명주기는 보통 다음 순서로 동작합니다.

1. 스프링 컨테이너가 빈을 생성합니다.
2. 의존관계를 주입합니다.
3. 초기화 작업을 수행합니다.
4. 빈이 사용됩니다.
5. 컨테이너 종료 시 소멸 작업을 수행합니다.

## 생명주기 콜백

스프링은 빈이 생성된 뒤 초기화 작업을 실행하거나, 소멸되기 직전에 정리 작업을 실행할 수 있도록 생명주기 콜백을 제공합니다.

생명주기 콜백은 객체 생성 직후 추가 설정이 필요할 때, 외부 시스템과 연결을 맺어야 할 때, 애플리케이션 종료 전에 자원을 반납해야 할 때 사용합니다.

스프링에서는 여러 방식으로 생명주기 콜백을 처리할 수 있습니다. 실무에서는 보통 **`@PostConstruct`와 `@PreDestroy`** 를 가장 많이 사용합니다. `InitializingBean`과 `DisposableBean`도 사용할 수 있지만, 스프링 인터페이스에 직접 의존하게 됩니다.

### @PostConstruct와 @PreDestroy

`@PostConstruct`는 빈이 생성되고 의존관계 주입이 끝난 뒤 호출되는 메서드에,`@PreDestroy`는 빈이 소멸되기 직전에 호출되는 메서드에 설정합니다.

아래 코드와 같이 사용할 수 있습니다.

```java
@Component
public class NetworkClient {

    @PostConstruct
    public void init() {
        System.out.println("초기화 작업 수행");
    }

    @PreDestroy
    public void close() {
        System.out.println("연결 종료");
    }
}
```

### InitializingBean과 DisposableBean

스프링은 `InitializingBean`과 `DisposableBean` 인터페이스도 제공합니다.

`InitializingBean`은 초기화 시 `afterPropertiesSet()` 메서드를 호출하고, `DisposableBean`은 소멸 시 `destroy()` 메서드를 호출합니다.

아래 코드와 같이 오버라이드하여 사용할 수 있습니다.

```java
@Component
public class NetworkClient implements InitializingBean, DisposableBean {

    @Override
    public void afterPropertiesSet() {
        System.out.println("초기화 작업 수행");
    }

    @Override
    public void destroy() {
        System.out.println("연결 종료");
    }
}
```

`InitializingBean`, `DisposableBean`을 사용하는 경우 빈 클래스가 인터페이스를 직접 구현해야 하므로 코드가 스프링에 의존하게 됩니다.

## 빈 스코프

빈 스코프는 스프링 컨테이너가 빈을 어떤 범위와 생명주기로 관리할지를 정하는 개념입니다.

같은 클래스를 빈으로 등록하더라도, 스코프에 따라 하나의 객체를 계속 재사용할 수도 있고, 요청이 들어올 때마다 새로 만들 수도 있습니다. 어떤 스코프를 선택하느냐에 따라 빈의 생성 시점, 공유 범위, 생명주기가 달라집니다.

스프링에서 자주 사용하는 스코프에는 싱글톤, 프로토타입, 웹 스코프가 있습니다.

### 싱글톤

싱글톤은 스프링의 기본 스코프입니다. 별도로 스코프를 지정하지 않으면 대부분의 빈은 싱글톤으로 등록됩니다.

싱글톤 스코프에서는 **스프링 컨테이너당 하나의 빈 인스턴스만 생성**합니다. 이후에는 같은 빈을 요청하더라도 같은 객체를 반환합니다.

서비스, 레포지토리, 컨트롤러처럼 애플리케이션 전반에서 공통으로 사용하는 객체는 보통 싱글톤으로 관리합니다. 이 방식은 객체를 반복해서 생성하지 않아도 되며, 애플리케이션 전체에서 공통 설정을 공유하기 쉽습니다.

다만 여러 요청이 같은 객체를 함께 사용할 수 있으므로, **요청별로 달라지는 상태를 필드에 저장하지 않도록** 주의해야 합니다. 따라서 싱글톤 빈은 보통 **상태를 가지지 않는 객체**로 설계합니다.

### 프로토타입

프로토타입 스코프는 **빈을 요청할 때마다 새로운 객체를 생성**하는 방식입니다.

싱글톤과 달리 컨테이너가 하나의 인스턴스를 계속 재사용하지 않습니다. 같은 빈을 여러 번 조회하면 매번 새로운 객체가 반환됩니다.

이 스코프는 **사용할 때마다 새로운 상태가 필요한 객체**에 적용할 수 있습니다. 매 사용마다 독립적인 데이터를 담아야 하는 객체라면 프로토타입 스코프가 더 적절할 수 있습니다.

다만 **프로토타입 빈은 생성과 초기화까지만 스프링 컨테이너가 관리**합니다. 그 이후 소멸 과정은 기본적으로 컨테이너가 관리하지 않으므로, 종료 시 소멸 작업이 필요하다면 개발자가 별도로 처리해야 합니다.

### 웹 스코프

웹 스코프는 웹 환경에서만 사용할 수 있는 스코프입니다. HTTP 요청이나 세션처럼 웹 애플리케이션의 범위에 맞춰 빈을 관리합니다.

대표적으로 다음과 같은 스코프가 있습니다.

- request: 하나의 HTTP 요청 동안만 빈을 유지합니다.
- session: 하나의 HTTP 세션 동안 빈을 유지합니다.
- application: 웹 애플리케이션 범위에서 빈을 유지합니다.
- websocket: 웹소켓 세션 범위에서 빈을 유지합니다.

예를 들어 request 스코프 빈은 요청이 시작될 때 생성되고 요청이 끝나면 함께 소멸됩니다. session 스코프 빈은 같은 사용자의 세션 동안 유지되다가 세션이 종료되면 함께 소멸됩니다.

---

싱글톤 빈은 스프링 컨테이너가 생성될 때 만들어지고 애플리케이션 전체에서 공유됩니다. 반면 request 스코프나 session 스코프 빈은 HTTP 요청이나 세션이 실제로 존재할 때 만들어집니다. 생성 시점과 유지 범위가 서로 다르기 때문에, **싱글톤 빈이 웹 스코프 빈을 그대로 주입받으면 문제가 생길 수 있습니다.**

예를 들어 싱글톤 서비스가 request 스코프 빈을 의존한다고 가정하면, 서비스 빈은 애플리케이션 시작 시점에 먼저 생성됩니다. 하지만 request 스코프 빈은 아직 HTTP 요청이 없기 때문에 생성할 수 없습니다. 이때 스프링은 실제 request 스코프 빈을 바로 주입하는 대신, **프록시 객체**를 주입할 수 있습니다.

프록시는 겉으로는 request 스코프 빈처럼 보이지만, 실제 객체 조회는 메서드가 호출되는 시점까지 하지 않습니다. 이후 HTTP 요청이 들어온 상태에서 해당 빈을 사용하면, 프록시가 현재 요청에 맞는 실제 request 스코프 빈을 찾아 호출합니다. 이렇게 하면 싱글톤 빈은 생성 시점에 request 스코프 빈이 없어도 의존관계를 가질 수 있고, 실제 사용 시점에는 요청별로 다른 빈을 사용할 수 있습니다.
